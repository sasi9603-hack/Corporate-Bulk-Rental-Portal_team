Corporate Bulk Rental Portal — System Analysis, Database Design & Security

1. Architecture & Technology Stack

Our project uses a modern Serverless / BaaS (Backend-as-a-Service) architecture. Unlike standard applications that require hosting and writing a separate backend server (e.g. Express/NodeJS or Spring Boot), we leverage Supabase to handle routing, authentication, API generation, and database interactions.

[ React Client (Vite + Tailwind) ] | | Secure REST/Websocket Queries (JWT Auth Headers) ▼ [ Supabase BaaS Gateway (Auto-generated APIs) ] ├── JWT Authentication & Verification ├── PostgreSQL Database Management └── RLS (Row-Level Security) Query Filter | ▼ [ PostgreSQL Database (7 Relational Tables) ]
💡 Why BaaS & Supabase?
  • Instant APIs: Supabase exposes endpoints directly for every SQL table we create, removing the need to write redundant controller routes.
  • Database-level Security: Standard servers use middleware. We write policies directly inside PostgreSQL using Row Level Security (RLS).
  • PostgreSQL Engine: Provides relational schemas, constraints, triggers, and transactions.

2. Database Design & Schema

Our database consists of 7 tables structured in a relational database layout to support the bulk booking process.

  • profiles: Extends Supabase auth details. Structured as: id (UUID, PK references auth.users), email (text), full_name (text), role (text check constraints: 'admin', 'client').
  • companies: Captures B2B details. Columns: id (UUID, PK), company_name, contact_person, email, phone, address.
  • devices: The rental inventory. Columns: id (UUID, PK), name, category (check: Laptop/Desktop/Monitor/Projector/Printer), available_quantity, daily_price.
  • rental_requests: Bookings submitted by clients. Columns: id (UUID, PK), company_id (FK -> companies), user_id (FK -> auth.users), event_name, start_date, end_date, delivery_location, status (Pending, Under Review, Quoted, Approved, Allocated, Delivered, Completed, Rejected).
  • request_items: Junction table mapping requests to requested devices. Columns: id (UUID, PK), request_id (FK), device_id (FK), quantity (integer).
  • quotations: Billing details per request. Columns: id (UUID, PK), request_id (FK, Unique), total_amount (numeric), quotation_notes (text), status (check: Draft/Sent/Approved/Rejected).
  • status_history: State-machine log auditing transitions. Columns: id (UUID, PK), request_id (FK), old_status, new_status, changed_by (FK), admin_note, changed_at.
⚠️ Integrity & Constraints

We use Foreign Key Constraints with ON DELETE CASCADE to prevent orphan rows. For example, deleting a rental request cascades deletions down to its child records in request_items and quotations automatically.

3. Row Level Security (RLS) & Security Policies

Every table in our database has Row Level Security enabled. RLS ensures that unless an explicit policy grants access, queries will return empty rows.

Supabase verifies client requests via JWT (JSON Web Tokens). The database extracts the client ID using the function auth.uid() and checks the access policies.

-- Example RLS Policy
CREATE POLICY "Clients can view own requests" ON rental_requests
  FOR SELECT USING (auth.uid() = user_id);

-- Custom Authorization Checker
CREATE OR REPLACE FUNCTION is_admin()
RETURNS BOOLEAN LANGUAGE sql SECURITY DEFINER AS $$
  SELECT EXISTS (
    SELECT 1 FROM profiles 
    WHERE id = auth.uid() AND role = 'admin'
  );
$$;

4. Mock Interview Q&As (15 Questions)

Q1: Which database did you use, and why?
We used PostgreSQL hosted on Supabase. PostgreSQL is an enterprise-grade relational database. It is ideal for transactional business portals because it enforces strict schemas, referential integrity (foreign keys), and secure transactions, preventing issues like orphan rows or double bookings.
Q2: How does your frontend talk to the database without a custom server (like Node/Express)?
We use Supabase as our Backend-as-a-Service (BaaS). Supabase automatically generates RESTful APIs directly from our PostgreSQL database tables. Our React client imports the @supabase/supabase-js SDK and queries the database directly via HTTPS calls, eliminating server-side routing boilerplate.
Q3: Explain how Row Level Security (RLS) protects client data.
RLS is a PostgreSQL security feature. Every time a client queries a table, the database checks defined RLS policies. It verifies the client’s identity using the JWT sent in the authorization header and extracts their user ID using auth.uid(). The policy restricts the query to return only rows matching that ID, e.g., auth.uid() = user_id.
Q4: How does client user registration work? How is their profile created?
When a user signs up via Supabase Auth, their email and password are saved in Supabase’s internal auth.users table. We set up a PostgreSQL Database Trigger (on_auth_user_created) that automatically executes a function. This function inserts a matching row into our public profiles table, mapping their metadata and assigning a default role of 'client'.
Q5: How do you handle role authorization (Admins vs. Clients)?
We use Role-Based Access Control (RBAC). The profiles table has a role column ('admin' or 'client'). On the database level, RLS policies use a custom function is_admin() to check this role and restrict modifications (like adding inventory or quotations). On the frontend, React uses route guards to check the role in AuthContext and redirect clients away from admin pages.
Q6: How is the rental request quotation calculated and exported?
The quotation total is calculated dynamically: daily_device_rate * quantity * event_duration_days. Once approved, the admin inputs this total, and we use the client-side library jsPDF to generate a formatted PDF invoice displaying order details and company branding, downloading it directly via the browser.
Q7: What happens to related records when a request is deleted?
We use foreign key constraints configured with ON DELETE CASCADE. For example, request_items and quotations have foreign keys referencing the rental_requests table. If an admin deletes a request, PostgreSQL automatically removes all associated items and quotes, protecting data consistency.
Q8: How does do you audit status changes for the rental request workflow?
We created a status_history table that functions as an audit log. Every time a request status updates (e.g., from Pending to Under Review to Quoted), the update trigger or transaction inserts a row documenting the old state, new state, user ID of the modifier, timestamp, and optional remarks.
Q9: What is the difference between Supabase Service Role key and Anon key?
The Anon Key is public and safe to compile in client-side code; it respects RLS rules. The Service Role Key is private, bypasses all RLS checks entirely, and must only be stored securely on backend systems (like Edge Functions or secure servers), never in the browser.
Q10: How do you secure database credentials in local development vs. production?
We store database URLs and keys inside a .env file, which is added to .gitignore so it is never pushed to GitHub. When deploying to production on Vercel, we add these keys directly into Vercel’s secure Dashboard Environment Variables.
Q11: How does the system handle real-time notifications?
Supabase utilizes PostgreSQL’s replication logs to broadcast events. The React client subscribes to these streams via WebSockets (supabase.channel()), listening for database events like inserts on rental_requests to update the Admin Dashboard instantly.
Q12: How do you handle email verification?
We integrated the Resend SMTP service. When a user registers, Supabase Auth intercepts the action and sends an verification email containing a JWT confirmation token. The user cannot access the client portal until they click the link and change their profile status to verified.
Q13: How are database tables optimized for rapid search queries?
We created indexes on foreign key columns (like request_id and user_id) that are frequently scanned during JOIN operations. This prevents slow sequential table scans as our request history table scales.
Q14: How does the database prevent bad data from being entered?
We use PostgreSQL check constraints on columns. For instance, the status column in rental_requests has a constraint limiting values to a specific state machine: check (status in ('Pending', 'Under Review', 'Quoted', ...)). Any insert violating this will throw an SQL database error.
Q15: Explain how transaction control works in your request form.
When a user submits a form, we need to create a company record, a request record, and multiple item records. If one insert fails, they must all fail to prevent partial data corruption. In Supabase, we achieve this by invoking a single PostgreSQL RPC (stored procedure) function that executes these steps inside a standard SQL BEGIN...COMMIT block.
